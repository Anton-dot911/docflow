# ТЗ №1 — Document Intelligence Pipeline

**Кодова назва:** DocFlow
**Тип:** портфоліо-продукт
**Пріоритет:** перший у черзі реалізації
**Оцінка тривалості:** 3 тижні (MVP — 2 тижні)

---

## 1. Проблема та бізнес-кейс

Малий і середній бізнес щодня отримує неструктуровані документи — рахунки-фактури, акти виконаних робіт, комерційні пропозиції, замовлення email-ом. Ручне перенесення даних у таблиці/облікові системи: повільне, з помилками, не масштабується.

**Рішення:** сервіс, що приймає документ (PDF, скан, фото, email), витягує структуровані дані за визначеною схемою, валідує їх і віддає готовий JSON/CSV або запис у БД — з інтерфейсом human-in-the-loop для перевірки сумнівних полів.

**Що демонструє роботодавцю:** повний цикл роботи з даними (ingestion → extraction → validation → review → export), structured outputs, обробку edge cases, вимірювану точність.

## 2. Користувачі та сценарії

- **Оператор** (бухгалтер, менеджер): завантажує пачку документів, переглядає результати, виправляє позначені поля, експортує.
- **Адміністратор:** налаштовує схеми екстракції (типи документів і їхні поля).
- **Рекрутер/замовник (демо-режим):** відкриває демо без реєстрації, завантажує зразок або обирає готовий приклад, бачить результат за секунди.

**Ключовий сценарій (happy path):** завантаження PDF рахунку → авто-визначення типу документа → екстракція полів → поля з упевненістю нижче порогу підсвічені → оператор підтверджує/править → експорт CSV.

## 3. Функціональні вимоги

### MVP (обов'язково)
- **F1. Завантаження:** PDF (текстовий і сканований), JPG/PNG. Drag-and-drop, до 10 файлів за раз, ліміт 10 МБ/файл.
- **F2. Класифікація документа:** авто-визначення типу (рахунок / акт / інше) перед екстракцією.
- **F3. Екстракція:** для кожного типу — фіксована Pydantic-схема. Для рахунку: постачальник (назва, ЄДРПОУ/ІПН), покупець, номер, дата, позиції (назва, к-сть, ціна, сума), разом, ПДВ, всього. Кожне поле — зі значенням `confidence` (0–1) і посиланням на фрагмент джерела.
- **F4. Валідація:** детерміновані перевірки після екстракції — арифметика позицій, формат дат, контрольні розряди ЄДРПОУ. Помилки валідації знижують confidence поля до 0.
- **F5. Review UI:** двопанельний інтерфейс — зліва документ, справа форма з полями; поля з confidence < 0.85 підсвічені; клік по полю підсвічує фрагмент у документі.
- **F6. Експорт:** JSON, CSV; збереження підтверджених записів у Postgres.
- **F7. Демо-режим:** 5 передзавантажених документів різної складності, доступ без реєстрації.

### v1.1 (після MVP)
- F8. Кастомні схеми: адміністратор описує новий тип документа через UI (назва поля, тип, обов'язковість) — без коду.
- F9. Email-ingestion: пересилання на службову адресу → авто-обробка вкладень.
- F10. Batch API для великих пакетів (зниження вартості).
- F11. Webhook після обробки (інтеграція у чужі системи).

### Поза межами (out of scope)
Рукописний текст; документи не укр/англ мовами; інтеграція з конкретними обліковими системами (BAS/1С).

## 4. Нефункціональні вимоги

- Обробка одного документа ≤ 20 с (p95).
- Structured-output validity rate ≥ 99% (схема завжди валідна).
- Field-level accuracy ≥ 95% на golden dataset (див. §8).
- Вартість обробки одного документа логована; ціль ≤ $0.03.
- Дані користувачів ізольовані (RLS у Supabase); файли видаляються за запитом.

## 5. Архітектура та стек

```
[React SPA] ──HTTPS──> [FastAPI]
                          ├─ /ingest: збереження файлу (Supabase Storage), черга
                          ├─ Worker: препроцесинг → класифікація → екстракція → валідація
                          ├─ Claude API (vision для сканів, structured outputs)
                          └─ Postgres (Supabase): documents, extractions, schemas, review_log
```

- **Frontend:** React + Vite + TypeScript + Tailwind (Netlify).
- **Backend:** Python 3.12, FastAPI, Pydantic v2, `anthropic` SDK. Деплой: Railway/Fly.io.
- **Черга:** для MVP — фонові задачі FastAPI (BackgroundTasks) + статуси в БД; polling з фронтенду. Redis/Celery — лише якщо з'явиться реальна потреба.
- **Препроцесинг PDF:** текстові PDF — витяг тексту (pypdfium2/pdfplumber); скани та зображення — передача сторінок як зображень у Claude (vision). Рішення "текст чи зображення" — за часткою витягнутого тексту.
- **Structured outputs:** екстракція через tool use зі схемою, згенерованою з Pydantic-моделі; відповідь валідовується тією ж моделлю. Одна повторна спроба з повідомленням про помилку валідації.

## 6. Модель даних (ядро)

- `document(id, user_id, filename, storage_path, doc_type, status[queued|processing|review|confirmed|failed], created_at)`
- `extraction(id, document_id, schema_version, payload jsonb, field_confidences jsonb, validation_errors jsonb, model, tokens_in, tokens_out, cost_usd, latency_ms)`
- `review_log(id, extraction_id, field, old_value, new_value, reviewer, ts)` — джерело даних для майбутнього поліпшення промптів.
- `extraction_schema(id, doc_type, version, json_schema, is_active)`

## 7. AI-компоненти

| Компонент | Вхід | Вихід | Промпт-файл |
|---|---|---|---|
| Класифікатор | 1-ша сторінка (текст/зображення) | `{doc_type, confidence}` | `prompts/classify.v1.md` |
| Екстрактор | повний документ + схема | payload за схемою + confidence по полях | `prompts/extract_invoice.v1.md` та ін. |

Правила: температура 0; у промпті екстрактора — 2 few-shot приклади складних кейсів (багатосторінковий рахунок, скан низької якості); інструкція повертати `null` замість вигадування значення.

## 8. Evals та якість

- **Golden dataset:** ≥ 40 документів (реальні знеособлені + синтетичні), розмічені через Golden Dataset Builder. Категорії: чистий текстовий PDF, скан хорошої якості, скан поганої якості, фото, багатосторінковий, нестандартний макет.
- **Метрики:** field accuracy (exact match для чисел/дат, fuzzy ≥ 0.9 для назв), schema validity rate, частка полів, відправлених на review, false-confidence rate (confidence ≥ 0.85, але значення хибне — найнебезпечніша помилка).
- **Скрипт:** `make eval` → таблиця метрик по категоріях + порівняння з попереднім запуском. Результати — у README.

## 9. Етапи реалізації

1. **Тиждень 1:** скаффолд (через Project Scaffolder), схема БД, ingestion, препроцесинг, екстракція для одного типу (рахунок), CLI-прогін на 10 документах.
2. **Тиждень 2:** Review UI, валідації, експорт, демо-режим, golden dataset (40 док.), перший eval-прогін, ітерація промптів до цільових метрик.
3. **Тиждень 3:** класифікатор + другий тип документа (акт), полірування UI, README, відео-демо, деплой, публікації.

## 10. Definition of Done

- [ ] Живе демо без реєстрації обробляє документ ≤ 20 с
- [ ] Field accuracy ≥ 95%, schema validity ≥ 99% на golden dataset — цифри в README
- [ ] Review-цикл повний: правка поля зберігається в review_log
- [ ] Усі виклики LLM видно в Cost/Latency Tracker
- [ ] README (EN) з архітектурною діаграмою і розділом Design decisions
- [ ] Відео-демо ≤ 2 хв

## 11. Ризики

| Ризик | Мітигація |
|---|---|
| Погані скани → низька точність | Категорія в eval-датасеті з окремим порогом; чесне повідомлення "needs review" замість вигадування |
| Вартість vision-запитів | Стиснення зображень до розумної роздільності; Batch API у v1.1 |
| Галюцинації у числових полях | Детермінована арифметична валідація (F4) ловить розбіжності |
